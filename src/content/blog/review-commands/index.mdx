I've been using two little shell functions to help me work with AI coding assistants: `review_start` and `review_done`.

They let me do a full review pass in my editor, leave inline comments and fixes, and then hand the agent a commit containing only my feedback.

## The idea

Suppose an AI agent has just made a bunch of changes on your current branch. Run:

```bash
review_start "implemented XYZ"
```

This:

1. commits the AI-generated changes on your current branch,
2. creates a temporary review branch,
3. resets that branch back one commit while keeping the changes in your working tree.

So the AI's work is persisted as a real commit, but in your editor it still looks like ordinary uncommitted changes.

The resulting history is roughly:

```text
A -- B -- C -- D   (my-feature)
           \
            C    (${USER}/review_in_progress working tree contains changes from D)
```

Here `D` is the AI snapshot commit. The review branch points at `C`, but your working tree still contains the diff from `D`, so you can review it naturally in VS Code or any other IDE.

While reviewing, I often:

- edit code directly,
- leave inline temporary comments like `TDC: factor this out into a helper`,
- ask questions like `TDC: why did you do X here?`

When I'm done, I run:

```bash
review_done
```

That switches back to the original branch, deletes the temporary review branch, and creates a new commit containing only my review edits and comments:

```text
A -- B -- C -- D -- E   (my-feature)
```

Now `E` is a precise “please address this review” commit. I can just tell the agent:

```bash
git show HEAD
```

and it can see exactly what I changed, without guessing and without clobbering my edits.

## Why I like this

The big win is that it turns “please address my review comments” into a concrete Git object.

- My requests are right next to the code I was looking at when I made the request.
- The agent can see what I already changed, so it is less likely to revert my manual fixes.
- It is token-efficient: I can hand over one commit instead of dribbling out comments one at a time.
- I do one deliberate review pass instead of losing track of what I've already looked at.

## Installation

Put `review_aliases.sh` somewhere in your dotfiles repo and source it from your `~/.bashrc`:

```bash
if [ -f "$HOME/path/to/review_aliases.sh" ]; then
  . "$HOME/path/to/review_aliases.sh"
fi
```

Then reload your shell:

```bash
source ~/.bashrc
```

## The workflow

### 1. Let the AI make changes

Start on your normal working branch and let the agent do its thing.

### 2. Snapshot the AI result

```bash
review_start "implemented XYZ"
```

This creates a commit on your branch, then moves you to `${USER}/review_in_progress` with the same diff present as working-tree changes.

### 3. Review in your editor

Open the Git diff UI in your editor and review as usual.

I use `TDC:` as a temporary marker for comments I want the AI to address later. By the end, there should be no `TDC:` left anywhere.

### 4. Finish the review

```bash
review_done
```

This commits only the delta introduced during your review.

### 5. Hand that commit back to the AI

```bash
git show HEAD
```

Now the agent can apply your feedback against a precise diff.

## Caveats

A few important details:

- The current version uses `git commit -am ...`, so it handles tracked modified/deleted files but not new untracked files. If the AI created new files, `git add` them first.
- Only one review session can be active per repo.
- `review_done` expects you to still be on the review branch.
