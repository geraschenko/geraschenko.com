I've been using two little shell functions to help me work efficiently with AI coding assistants: `review_start` and `review_done`. These allow me to see all changes made by the coding agent and leave inline comments/changes, but keep my changes _separate_ from the agents changes. This means that after a review pass, I can easily point the agent to just the stuff I changed.

## The idea

Suppose an AI agent has just made a bunch of changes on your current branch.

Run:

```bash
review_start "implemented XYZ"
```

This:

1. commits the AI-generated changes on your current branch,
2. creates a temporary review branch,
3. resets that branch back one commit while keeping the changes in your working tree.

So now you have the best of both worlds:

- the AI's work is safely persisted as a real commit,
- but in your editor it still looks like ordinary uncommitted changes.

That means you can open VS Code (or whatever IDE), go to the Git changes view, and review the whole patch naturally.

While reviewing, I often:

- edit code directly,
- leave inline temporary comments like `TDC: factor this out into a helper` for the AI to address.
- ask questions like `TDC: why did you do X here?`

When I'm done, I run:

```bash
review_done
```

This switches back to the original branch, deletes the temporary review branch, and creates a new commit containing only my review edits and comments.

## Why I like this

The big win is that it turns “please address my review comments” into a concrete Git object that I can hand to the agent.

* All the requests for changes are right next to the code I was looking at when I left the request. 
* It's aware of things I changed so it doesn't revert my manual fixes.
* All this takes a single conversational turn with the agent, so it's much more token-efficient than providing the comments/requests one at a time.
* I don't get confused about what I have or haven't already reviewed. (But the flip side is that I *have* to do a full review pass before running `review_done`)

## Installation

Put `review_aliases.sh` somewhere in your dotfiles repo and source it from your `~/.bashrc`:

```bash
if [ -f "$HOME/path/to/review_aliases.sh" ]; then
  . "$HOME/path/to/review_aliases.sh"
fi
```

Then reload your shell:

```bash
source ~/.bashrc
```

## The workflow

### 1. Let the AI make changes

Start on your normal working branch and let the agent do its thing.

### 2. Snapshot the AI result

```bash
review_start "implemented XYZ"
```

This creates a commit on your branch, then moves you to `${USER}/review_in_progress` with the same diff present as working-tree changes.

### 3. Review in your editor

Open the Git diff UI in your editor and review as usual.

I use `TDC:` as a temporary marker for comments I want the AI to address later. By the end, there should be no `TDC:` left anywhere.

### 4. Finish the review

```bash
review_done
```

This commits only the delta introduced during your review.

### 5. Hand that commit back to the AI

```bash
git show HEAD
```

Now the agent can apply your feedback against a precise diff.

## Caveats

A few important details:

- The current version uses `git commit -am ...`, so it handles tracked modified/deleted files but not new untracked files. If the AI created new files, `git add` them first.
- Only one review session can be active per repo.
- `review_done` expects you to still be on the review branch.

## The script

The shell helpers used for this post are in `review_aliases.sh` next to this article in the repository.

If you want the short version, it is just this:

review_done
# then tell the AI: git show HEAD
